## 1. Where we are

Phase 1 and Phase 1b are merged on `main`:

| Phase | What shipped | Files |
|---|---|---|
| **1**  | `ClusterHealthAutopilot` CRD types, pure-function builders, CRD shipped via chart | `api/v1alpha1/`, `internal/operator/builders.go`, `charts/.../templates/crd-clusterhealthautopilot.yaml` |
| **1b** | Reconciler, `cmd/cha-operator/main.go` manager binary, chart's `operator-deployment.yaml` gated on `operator.enabled` (default off), 11 reconciler unit tests | `internal/operator/reconciler.go`, `cmd/cha-operator/`, `charts/.../templates/operator-deployment.yaml` |

What today's reconciler owns (per `reconciler.go`):
  - `ServiceAccount`, watcher `Deployment`, diagnose `CronJob`, remediate `CronJob`.
  - `Ready` + `WatcherRunning` status conditions.
  - **Owner references** on each child (cascade-delete when the CR is deleted).
  - **CreateOrUpdate** semantics (NOT server-side apply) so chart-managed installs stay un-touched until an operator explicitly creates a CR.

What today's reconciler does NOT touch:
  - **Cluster-wide reader `ClusterRole` + `ClusterRoleBinding`** — still chart-managed (`templates/clusterrole-watcher.yaml`, `templates/clusterrolebinding-watcher.yaml`).
  - **`ResolutionRecord` CRD + its ClusterRole** — chart-managed (P1/G4 work, lives outside the operator's reconcile loop).
  - **AI-stack resources** — aiwatch Deployment, approval-server Deployment, RAG Qdrant StatefulSet, approval ConfigMaps + their RBAC. All chart-managed.

Phase 1c closes the first of those gaps (cluster-wide reader RBAC) and adds the packaging layer (OLM bundle) so operators can ship CHA through OperatorHub eventually.

## 2. Phase 1c scope

**Explicitly in scope:**
1. **Operator-provisioned reader `ClusterRoleBinding`** for the watcher SA, written by the controller when it reconciles a `ClusterHealthAutopilot` CR (so an operator-managed install creates the binding without the chart having to ship one).
2. **Reader `ClusterRole`** — single shared cluster-scoped role, NOT owned by any CR (a `ClusterRole` is cluster-scoped; `OwnerReferences` only cascade for in-namespace owners). Created idempotently on first reconcile; survives CR delete.
3. **OLM bundle scaffolding** — `operator-sdk generate bundle` output committed under `bundle/`, `ClusterServiceVersion` (CSV) authored, install strategy = `Deployment` matching the chart's operator-deployment shape.
4. **Smoke-tested with OLM on minikube** — CSV applies clean, operator deploys, a `ClusterHealthAutopilot` CR reconciles end-to-end.
5. **CHANGELOG + design-doc cross-links.**

**Explicitly OUT of Phase 1c** (deferred to Phase 2 / 3 / 4):
  - **AI-stack reconcile** (aiwatch, approval-server, RAG Qdrant, approval ConfigMap stores) — these stay chart-managed. They were finalized in P2/P3/P4 of the gap-closure program and have their own values-surface; folding them into the CR is a Phase 2 design decision (do they get sub-resources like `AIPolicy`, or just nested `spec.ai.*` like helm?).
  - **Webhook validation** — Phase 2.
  - **`AlertmanagerConnected` condition** — Phase 2.
  - **OperatorHub submission** — Phase 4.
  - **Multi-cluster `ApplicationSet`** — Phase 4 (interim pattern already documented in the operator-plan memory).

The Phase 1c diff should land in a single PR: ~one file each for the new reconcile method + RBAC builder + bundle directory + unit tests.

## 3. CR shapes

### 3.1 `ClusterHealthAutopilot` — what's there today (Phase 1)

Mirrors the helm `values.yaml` so an operator-managed install has the same surface as a chart-managed install. Current top-level Spec fields (`api/v1alpha1/clusterhealthautopilot_types.go`):

```yaml
apiVersion: cha.bionicaisolutions.com/v1alpha1
kind: ClusterHealthAutopilot
metadata:
  name: bionic-cluster                  # one per namespace
  namespace: cluster-health-autopilot
spec:
  image: { repository, tag, pullPolicy, pullSecrets }   # required
  watcher:   { enabled, replicas, debounce, resyncPeriod }  # optional
  diagnose:  { enabled, schedule, backoffLimit, activeDeadlineSeconds }
  remediate: { enabled, schedule, dryRun }
  alerting:  { slack: { alerts, critical, healthInfo }, alertmanager: { url, clusterName } }
  serviceAccountName: ""                # operator names the SA when omitted
status:
  observedGeneration: <int>
  conditions:
    - type: Ready            { True | False | Unknown }
    - type: WatcherRunning   { True | False | Unknown }
```

Resource-scope: `Namespaced` (one CR per install per namespace). `shortName: cha` — `kubectl get cha` works as a routine query.

### 3.2 What Phase 1c adds to the CR shape

**Nothing in the Spec** — Phase 1c is about *what the controller reconciles*, not what fields the user sets. The new reader-RBAC is an implementation detail; the user doesn't configure it.

**One status condition is added:**

```go
// ConditionReaderRBACReady is set True when the operator-managed
// reader ClusterRole + ClusterRoleBinding both exist and bind to the
// CR's ServiceAccount. False = the cluster-wide read perms the
// watcher needs to do its job aren't fully provisioned.
ConditionReaderRBACReady = "ReaderRBACReady"
```

`Ready` is the AND of `WatcherRunning AND ReaderRBACReady` once Phase 1c lands.

### 3.3 Coexistence with `ResolutionRecord`

`ResolutionRecord` is a separate CRD (`crd-resolutionrecord.yaml`) introduced in P1/G4. It's the system-of-record for verified remediation outcomes — written by the watcher binary's `internal/resolution/recorder.go`, NOT by the operator.

**The operator does not own `ResolutionRecord`s.** It treats them like any other cluster resource: it can `list` them (the operator-deployment's `ClusterRole` may grant read access if we ever want status surfacing), but it doesn't create / delete / reconcile them. They're owned by the **diagnostic Subject** (`metadata.ownerReferences`) — typically the namespace or, in v1alpha2+, the offending resource itself.

The operator's CRD bundle ships BOTH CRDs for OLM packaging (so the bundle install registers both), but the controller only watches `ClusterHealthAutopilot`. This is the **standard OLM pattern**: bundle = one operator + N CRDs it understands; "understanding" can be passive (just registers the schema) or active (reconciles).

## 4. Reconcile loop boundaries

Phase 1b's reconciler is a lifecycle manager. The probe / analyze / fix loop runs inside the watcher pod (`cmd/cha`), NOT inside the controller. Phase 1c keeps that boundary; nothing in 1c moves logic into the reconcile path.

### 4.1 What the operator OWNS (after Phase 1c)

Per CR:
  - `ServiceAccount` (namespaced, ownerRef → CR)
  - `Deployment` (watcher, namespaced, ownerRef → CR)
  - `CronJob` × 2 (diagnose + remediate, namespaced, ownerRef → CR)

Cluster-scoped, shared across all CRs (Phase 1c new):
  - **`ClusterRole`** with the reader rules (one per cluster, idempotent; survives CR delete).
  - **`ClusterRoleBinding`** per CR's SA → the shared ClusterRole. Cluster-scoped resources can't carry an in-namespace owner-ref, so the binding is **cleaned up by the controller's finalizer** when the CR is deleted.

### 4.2 What the operator LEAVES ALONE

  - The chart's reader `ClusterRole` + `ClusterRoleBinding` (`templates/clusterrole-watcher.yaml`, etc.) when present. If a CR's SA already has read perms via a chart-managed binding, the operator's binding is **additive** (RBAC is permissive — multiple bindings union); no override, no race.
  - All AI-stack resources (`aiwatch-deployment.yaml`, `approval-server-deployment.yaml`, `rag-qdrant-statefulset.yaml`, approval ConfigMaps + their RBAC). The operator never touches them in 1c.
  - `ResolutionRecord` CRs (data, not control plane).
  - Any resource missing the `cha.bionicaisolutions.com/managed-by-cr: <name>` label that 1c starts setting (the operator only patches resources it labeled itself — defense in depth so a misconfigured CR can't claim a chart-managed Deployment).

### 4.3 Reconcile order (in one pass)

```
1. Fetch CR. If missing → return (finalizer handler ran).
2. If CR has DeletionTimestamp:
   - Remove ClusterRoleBinding for this CR's SA.
   - Remove finalizer. Return.
3. Ensure finalizer is present (so step-2 cleanup is guaranteed).
4. Validate spec (existing: image.tag non-empty).
5. ClusterRole (shared): CreateOrUpdate idempotently. Never delete.
6. ClusterRoleBinding (per-CR): CreateOrUpdate. Subject = CR's SA in CR's ns.
7. ServiceAccount → Deployment → CronJob × 2 (existing 1b flow).
8. Compute conditions:
   - WatcherRunning ← Deployment.Status.AvailableReplicas > 0
   - ReaderRBACReady ← both RBAC objects exist + binding targets the SA
   - Ready ← AND of the two
9. Patch status (deferred — happens even on mid-reconcile error).
```

Reconcile is **idempotent**: replaying the same spec produces the same cluster state, including a stable resourceVersion ladder (controller-runtime's CreateOrUpdate is a no-op when the desired matches observed).

### 4.4 Why a finalizer, not just owner-refs

`ClusterRoleBinding` is cluster-scoped. Kubernetes' garbage collector does not honor owner-refs from a namespaced object on a cluster-scoped child — the GC log line is `OwnerRef is namespaced, dropping`. So when the CR is deleted, K8s would leave the binding orphaned.

Phase 1c adds a finalizer (`cha.bionicaisolutions.com/operator-rbac`) so the controller gets a chance to delete the binding before the CR is GC'd. Standard controller-runtime pattern; no exotic logic.

## 5. Operator-provisioned reader RBAC — design

### 5.1 Shape

```yaml
# Cluster-scoped, one per cluster. Shared by every ClusterHealthAutopilot.
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cha-operator-watcher-reader
  labels:
    app.kubernetes.io/managed-by: cha-operator
rules:
  # Same surface as the existing chart's clusterrole-watcher.yaml.
  - apiGroups: [""]
    resources: ["pods", "services", "configmaps", "secrets", "events",
                "persistentvolumes", "persistentvolumeclaims", "nodes",
                "namespaces", "serviceaccounts"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    resources: ["deployments", "statefulsets", "daemonsets", "replicasets"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["batch"]
    resources: ["jobs", "cronjobs"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["autoscaling"]
    resources: ["horizontalpodautoscalers"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["networking.k8s.io"]
    resources: ["ingresses", "networkpolicies"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["cert-manager.io"]
    resources: ["certificaterequests", "certificates"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["acme.cert-manager.io"]
    resources: ["orders", "challenges"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["cha.bionicaisolutions.com"]
    resources: ["resolutionrecords"]
    verbs: ["get", "list", "watch"]  # the watcher reads its own history

---
# Per-CR.
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: cha-operator-watcher-<cr-namespace>-<cr-name>
  labels:
    app.kubernetes.io/managed-by: cha-operator
    cha.bionicaisolutions.com/managed-by-cr: <cr-name>
    cha.bionicaisolutions.com/cr-namespace: <cr-namespace>
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: cha-operator-watcher-reader
subjects:
  - kind: ServiceAccount
    name: <serviceAccountName resolved from CR>
    namespace: <cr-namespace>
```

The ClusterRole's verb set MUST exactly match `templates/clusterrole-watcher.yaml`. The operator-managed install should behave identically to a chart-managed install; any verb-set drift between the two managers is a regression worth a CI gate.

### 5.2 What if a chart-managed binding already exists?

Two installs side-by-side (chart-managed + operator-managed in the same cluster) yield two bindings targeting the same SA. RBAC unions across bindings, so this is harmless. The operator detects the chart-managed binding by its `helm.sh/chart` label and **does not delete** it — leaving the chart's lifecycle untouched is the v1.x compatibility contract.

When the chart-managed install is later removed (helm uninstall), the chart's binding goes with it; the operator's stays. Symmetric on the other side.

### 5.3 Builder + Reconcile new methods

```go
// internal/operator/builders.go — additions:
func BuildReaderClusterRole() *rbacv1.ClusterRole         // shared, no CR input
func BuildReaderClusterRoleBinding(cha *v1a1.ClusterHealthAutopilot) *rbacv1.ClusterRoleBinding

// internal/operator/reconciler.go — additions:
func (r *Reconciler) reconcileReaderRBAC(ctx context.Context, cha *v1a1.ClusterHealthAutopilot) error
func (r *Reconciler) finalizeReaderRBAC(ctx context.Context, cha *v1a1.ClusterHealthAutopilot) error
```

Tests follow the existing pattern in `reconciler_test.go` — fake-client driven, asserts the binding is created, that a second reconcile is a no-op, and that the finalize path deletes the binding.

## 6. OLM bundle

### 6.1 Layout

Generated by `operator-sdk generate bundle --version <vX.Y.Z>` and committed under `bundle/`:

```
bundle/
  manifests/
    cha.clusterserviceversion.yaml          # CSV
    cha.bionicaisolutions.com_clusterhealthautopilots.yaml
    cha.bionicaisolutions.com_resolutionrecords.yaml
  metadata/
    annotations.yaml                        # bundle metadata for OLM
  tests/scorecard/config.yaml               # operator-sdk scorecard config
bundle.Dockerfile
```

### 6.2 CSV — the install strategy

The CSV's `spec.install.strategy` mirrors the chart's `operator-deployment.yaml` 1:1 (same image, args, env, resources, securityContext, leader-election lease). The CSV references both CRDs as **owned** (the operator's domain) and declares `replaces` once we have a prior CSV version (Phase 1c is initial; no `replaces`).

`spec.installModes`:
  - `OwnNamespace: true`
  - `SingleNamespace: true`
  - `MultiNamespace: false` (Phase 1 reconciler doesn't watch multiple namespaces explicitly — it watches all via the manager's default scope; this can be tightened in Phase 2)
  - `AllNamespaces: true`

Categories: `Monitoring`, `OpenShift Optional` (so the bundle is OperatorHub-eligible in Phase 4).

### 6.3 Smoke-test in CI

A new GitHub Actions job (`bundle-smoke`) runs `operator-sdk run bundle <image>` against a kind cluster, creates a minimal `ClusterHealthAutopilot` CR, asserts the operator deploys + the CR's status reaches `Ready=True`. Adds ~5 minutes to CI on tag-push only — gated by a path filter so chart-only PRs don't trigger it.

### 6.4 Why not OLM in Phase 1b?

OLM coupling has real cost: `operator-sdk` brings its own opinions about leader-election + metrics endpoints + healthz that differ slightly from controller-runtime's defaults. Phase 1b chose plain controller-runtime + a chart-installed Deployment specifically to keep the OLM choice reversible. Phase 1c reconciles that — by Phase 1c the controller is stable, the chart wiring is proven, and adding the OLM layer is pure packaging.

## 7. Migration: chart → operator

| Path | Today | Phase 1c |
|---|---|---|
| **Greenfield install** | `helm install cha …` | `kubectl apply -f cha.yaml` (operator from OLM bundle reconciles the CR into the same workloads) |
| **Existing chart install** | Untouched | Untouched. Operator binding lands alongside the chart's binding (RBAC unions) — coexistence by default. |
| **Helm → operator cutover** | n/a | Documented runbook: (1) install operator + bundle; (2) `kubectl apply` the CR with `spec.serviceAccountName` set to the helm-created SA name; (3) `helm uninstall --keep-history` (which leaves the SA + RBAC objects bound to the operator-managed Deployment, since helm uninstall on hooks-only resources doesn't touch the SA helm transferred). The operator's reconcile adopts the chart-named workloads on first pass if the labels match. |

The cutover path is **opt-in and documented, not automatic** — Phase 1c does not ship migration tooling. The operator-plan memory notes the multi-cluster ApplicationSet path explicitly remains the interim solution until Phase 4.

## 8. Open questions before code starts

These are decisions the next sprint's owner has to make before opening the PR. They're documented here so the design conversation happens once, not in PR review.

1. **`AIPolicy` as a separate CRD vs `spec.ai.*` on `ClusterHealthAutopilot`?**
   The chart today has `ai.*` and `approval.*` as separate top-level values blocks because they have separate lifecycles (you can enable approval without AI being on, etc.). The CR-shape could mirror that (`ClusterHealthAutopilot.spec.ai` + `.spec.approval`) OR split into a sibling `AIPolicy` CRD with its own lifecycle. Recommendation: **start with nested fields in 1c**, split into `AIPolicy` only when there's a concrete need for independent lifecycle (e.g. operator wants to swap AI policy mid-install without re-rolling the watcher). Phase 1c leaves this untouched — AI stays chart-managed.

2. **Where does `ResolutionRecord` ownerRef point in v1alpha2?**
   Today the CR sets owner = the Subject's namespace (so cluster cleanup garbages records when a namespace dies). For v1alpha2, should records own a back-ref to the `ClusterHealthAutopilot` CR that produced them? **Recommendation: no** — the records are written by the watcher binary, not the operator; tying them to the operator's lifecycle would couple data retention to operator restarts. Records survive operator/CR replacement by design.

3. **Reader ClusterRole — what's the verb floor when the AI stack is enabled?**
   The AI stack adds: cert-manager `delete` on CertificateRequests, batch `delete` on Jobs, core `delete` on Pods, apps `patch` on Deployments. Those are *mutator* verbs and live on the **remediator** ClusterRole, not the reader. Phase 1c's reader is read-only — same as today. The mutator role is separately chart-managed and Phase 1c does not touch it. If we want operator-provisioned mutator RBAC later, that's a separate per-CR `ClusterRoleBinding` to the existing `<name>-remediator` ClusterRole.

4. **One ClusterRole per cluster vs one per CR?**
   Recommendation: **one per cluster, shared**. The verb set is identical across installs and decoupling them would just mean ClusterRole edits have to be replayed N times. The per-CR object is the binding. This matches how `system:` ClusterRoles work.

5. **Should the operator reconcile the chart's existing operator-managed resources (the Deployment running it)?**
   No — the operator's own Deployment is installed by OLM (in 1c+) or by the chart (in 1b). Self-managing the operator deployment is bootstrapping-with-eggs territory; standard OLM pattern is to let OLM (or helm) own the operator.

6. **What's the version-tag relationship between the operator + the watcher binary?**
   Same Dockerfile builds both today (`cha` + `cha-operator`). Phase 1c keeps that. The CSV pins a specific image tag, and the controller defaults `spec.image.tag` to its own tag when the CR omits one (operator self-knowledge — already in the Phase 1 builder). A user can override via the CR spec to test a newer watcher against an older operator, but the supported matrix is "same tag for both."

## 10. Acceptance criteria

Phase 1c is done when:

  - [ ] `kubectl apply -f cha.yaml` against a fresh kind cluster with the operator running produces a working watcher, with the operator-provisioned `ClusterRole` + `ClusterRoleBinding` granting the SA cluster-wide read.
  - [ ] Deleting the CR removes the binding (finalizer fires); the shared ClusterRole survives.
  - [ ] `operator-sdk run bundle docker4zerocool/cha-operator-bundle:v1.9.x` installs the operator + both CRDs via OLM on minikube.
  - [ ] The CSV passes `operator-sdk bundle validate ./bundle` cleanly.
  - [ ] `ConditionReaderRBACReady` flips to `True` once both RBAC objects are observed; `Ready` is the AND with `WatcherRunning`.
  - [ ] Side-by-side with a chart-managed install in the same cluster, no resource churn (the operator-managed binding is additive; the chart-managed binding stays helm-owned).
  - [ ] `make test` includes new reconciler-RBAC unit tests; coverage on `internal/operator/` doesn't regress.
  - [ ] CHANGELOG `Unreleased` block notes the Phase 1c additions; this design doc is referenced.
