# Operator port Phase 1 — Foundations (2026-05-27)

**Status:** Phase 1 (foundations) shipped — see CHANGELOG `Unreleased`.
**Parent design:** `docs/design/2026-05-v1.8-roadmap.md` §A1.
**Driver:** `project_srenix_operator_plan` memory — phased plan to evolve Srenix
from a pure Helm install into an in-cluster operator while keeping the
existing chart-managed installs working end-to-end.

## What Phase 1 ships

The first sub-phase of the operator port is **foundations only** — the
shape of the operator surface lands in `main` with zero impact on
existing chart-managed installs. The reconcile loop, the manager
binary, and the chart wiring that lets operators flip to
operator-managed installs all come in Phase 1b (next PR).

Concretely, Phase 1 adds:

  - **`api/v1alpha1/`** — Go types for `AgenticSRE`,
    `AgenticSREList`, and their `Spec`/`Status` shapes
    (`groupversion_info.go`, `agenticsre_types.go`).
    Hand-written `zz_generated.deepcopy.go` so the package builds
    without the codegen toolchain in CI.
  - **`internal/operator/`** — pure-function builders that translate
    a `AgenticSRESpec` into the concrete K8s manifests
    the controller will reconcile toward (`BuildServiceAccount`,
    `BuildWatcherDeployment`, `BuildDiagnoseCronJob`,
    `BuildRemediateCronJob`). 19 unit tests cover defaults,
    overrides, image-policy inference, pull-secret round-trip, and
    alerting-flag emission.
  - **`charts/.../templates/crd-agenticsre.yaml`** — the
    CRD itself, shipped via the existing chart so the resource type
    is registered on installs that adopt the v1.8 chart. Gated
    behind `operator.installCRD` (default `true`).

## What Phase 1 deliberately does *not* ship

  - **No `controller-runtime` dependency.** Adding it brings ~50 MB
    of transitive deps; reviewers should see that ground separately
    from the foundation types. Phase 1b adds it.
  - **No `cmd/srenix-operator/main.go`.** The manager binary lands in
    Phase 1b alongside the Reconcile loop.
  - **No envtest.** Builders are pure functions and tested as such;
    envtest's overhead lands when there's actually a controller to
    test.
  - **No chart-managed operator Deployment.** Phase 1c adds an
    operator-deployment template gated behind `operator.enabled`.

This staging means a v1.8.0-alpha release tag can be cut against
Phase 1 (CRD-only) without operators changing how they install Srenix.

## Why hand-written DeepCopy

`controller-gen` is the standard tool to generate DeepCopy boilerplate
from struct tags. Phase 1 deliberately avoids adding it to the build
because:

  1. The Spec is small (≤ 10 nested types). Hand-writing the methods
     is a one-time ~100-line cost.
  2. Adding `controller-gen` to CI means another binary to pin, plus
     `go install` overhead on every CI run.
  3. Phase 1b will already need `controller-runtime` for the
     reconcile loop. At that point regenerating
     `zz_generated.deepcopy.go` with `controller-gen` is a no-op
     diff — the hand-written file's function signatures are exactly
     the codegen output.

When Phase 1b lands, the hand-written file is replaced by
codegen output. The conservative choice now keeps this PR's diff
focused on the types themselves.

## Why ship the CRD ahead of the controller

Installing the CRD on a cluster that has no controller running is
harmless: the resource is just queryable state with no controller
acting on it. Shipping the CRD in Phase 1 means:

  - Existing chart installs that pull v1.8 chart immediately have
    the CRD registered, so the apiserver returns 200 for
    `kubectl apply -f srenix.yaml` rather than "no resource type".
  - Phase 1b can land the controller without a chart bump.
  - Operators who want to start authoring `AgenticSRE`
    manifests in their GitOps repo for review can do so against a
    live API before the controller ships.

## Builder design notes

  - **Pure functions, no controller dependencies.** The builders
    take a CR pointer and return a K8s manifest. No client-go, no
    informer, no Reconcile context. This makes them trivially
    testable and reusable in any future tool (e.g. a `srenix
    operator render` CLI subcommand that prints what a CR would
    produce).
  - **Mirror the chart's argument format.** The watcher binary
    already accepts `--alertmanager-url`, `--cluster-name`,
    `--slack-*-webhook-secret`, etc. The builders emit those exact
    flags, so the controller-managed install behaves identically to
    the chart-managed install. No `srenix` binary change required.
  - **Image-policy inference matches the chart helper.** The chart
    has a `srenix.pullPolicy` helper that picks `Always` for mutable
    tags (`latest`, `main`, `dev`) and `IfNotPresent` for semver.
    The builder's `pullPolicy()` does the same.
  - **`ServiceAccountName` override is honored everywhere.** An
    operator can point all Srenix workloads at a pre-existing SA by
    setting `spec.serviceAccountName`. Both the watcher Deployment
    and the CronJobs read through `ServiceAccountNameFor(cr)`.

## Acceptance criteria for Phase 1

  - [x] `go build ./...` passes.
  - [x] `go test ./api/... ./internal/operator/...` passes (19 tests).
  - [x] `helm template charts/agentic-sre` includes the
    CRD when `operator.installCRD=true`.
  - [x] `helm template charts/agentic-sre
    --set operator.installCRD=false` omits the CRD.
  - [x] Existing `helm unittest charts/agentic-sre`
    suite still passes 19/19.
  - [x] CHANGELOG `Unreleased` documents the Phase 1 scope and
    explicit Phase 1b deferral.

## Phase 1b scope (next PR)

  - Add `sigs.k8s.io/controller-runtime` to `go.mod`.
  - Replace hand-written `zz_generated.deepcopy.go` with
    `controller-gen` output (no-op diff if the hand-written body is
    correct).
  - Add `cmd/srenix-operator/main.go` — manager binary with
    leader-election (mirror the watcher's existing lease config),
    metrics endpoint, and the `AgenticSREReconciler`
    registration.
  - Add `internal/operator/reconciler.go` — the Reconcile method
    that:
    1. Fetches the CR.
    2. Calls `BuildServiceAccount` + `BuildWatcherDeployment` +
       `BuildDiagnoseCronJob` + `BuildRemediateCronJob`.
    3. Server-side-applies each.
    4. Updates `status.conditions` with `Ready` + `WatcherRunning`.
  - Add envtest-driven controller tests.
  - Add operator-deployment Helm template gated behind
    `operator.enabled`.

## Phase 1c scope

  - OLM bundle (`operator-sdk generate bundle`).
  - CSV (`ClusterServiceVersion`) with install strategy.
  - Helm repository alongside the OLM bundle for operators who
    prefer Helm-managed operator installs.
