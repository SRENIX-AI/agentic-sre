## Sequencing principle — two parallel streams

The operator-plan memory is explicit that Phase 1 is a **lifecycle manager
only**: the new controller reconciles the watcher `Deployment` + diagnose
`CronJob` + RBAC; the probe / analyze / fix loop stays in the existing
`cmd/cha` binary. That means new drift classes and cloud probes ship into
the existing packages and the operator picks them up automatically — no
merge-conflict serialisation needed.

- **Stream A** (drift classes + M2 K8s probes + operator Phase 1) — Go work
  in `internal/diagnose/` and `internal/probe/`, plus the new `controllers/`
  directory.
- **Stream B** (GCP + Azure cloud probes) — Go work in `pkg/cloud/{gcp,azure}/`
  and `internal/cloud/{gcp,azure}/`. Mirrors the existing
  `internal/cloud/aws/` shape (`acm_probe.go`, `alb_probe.go`,
  `ebs_probe.go`, `eks_*`, `iam_roles_probe.go`, `kms_probe.go`,
  `rds_probe.go`, `s3_public_access_probe.go`, `vpc_subnets_probe.go`).

## Workstream A — Operator port + remaining drift classes + M2 K8s probes

### A1. Operator port (Phase 1)

  - `kubebuilder init --domain cha.bionicaisolutions.com --repo github.com/Bionic-AI-Solutions/cluster-health-autopilot`
  - `kubebuilder create api --group cha --version v1alpha1 --kind ClusterHealthAutopilot`
  - Controller reconciles watcher `Deployment` + diagnose `CronJob` + RBAC
    (replaces the corresponding Helm templates' runtime decision-making;
    the chart stays as the operator's own deployment vehicle).
  - Status conditions: `Ready`, `WatcherRunning`.
  - Unit tests for controller reconcile (envtest).
  - Helm chart stays alongside; the operator is the new install surface
    for v1.8+. **No breaking change for existing pilots** — the chart still
    works.
  - **Target:** Sprint 1 close (2026-06-10). **Effort:** 5 days.

### A2. B4 — Config drift analyzer

  - `internal/diagnose/config_drift.go` — new.
  - ConfigMap content-hash differs across replicas (rolling update never
    completed, or someone `kubectl edit`'d a single replica).
  - CRD version mismatch between cluster-installed CRDs and the version
    referenced in DriftReport custom resources.
  - Helm release values vs cluster-live (overlap with B1 at the
    individual-resource level).
  - `analyzers.configDrift.enabled` Helm value + `CHA_ANALYZER_CONFIG_DRIFT=off`
    env-gate.
  - **Target:** Sprint 1 close. **Effort:** 4 days.

### A3. B5 — Capacity drift analyzer

  - `internal/diagnose/capacity_drift.go` — new.
  - Pod resource-request vs actual-usage divergence (idle waste / OOM risk).
  - PVC growth-trajectory: linear-fit on PVC fill-rate; flag if free space
    drops below 14 days at current rate.
  - HPA min/max divergence: pinned at `minReplicas` for > 30 days (not
    load-driven); pinned at `maxReplicas` (under-provisioned).
  - **Pre-flight check:** `metrics-server` availability — if missing, the
    analyzer no-ops and emits a single `MetricsServerMissing` warning so
    the operator notices. Don't fail the watcher.
  - `analyzers.capacityDrift.enabled` Helm value + env-gate.
  - **Target:** Sprint 2 close (2026-06-24). **Effort:** 5 days.

### A4. B6 — Security drift analyzer

  - `internal/diagnose/security_drift.go` — new.
  - Pod Security Standards downgrade: namespaces previously at `restricted`
    slipping to `baseline` / `privileged`.
  - Missing image attestation: workloads running images without a Cosign /
    Notation signature when a policy requires one.
  - NetworkPolicy coverage gaps in production namespaces (the policy-gap
    class specifically; complements B3's wildcard-verb check).
  - `analyzers.securityDrift.enabled` Helm value + env-gate.
  - **Target:** Sprint 3 close (2026-07-08). **Effort:** 4 days.

### A5. M2 K8s probes (4 × 2-day probes)

  - **Kong** route / upstream health probe (`internal/probe/kong_probe.go`).
  - **HPA** scaling-failure probe (HPA reports `ScalingActive=False` or
    `AbleToScale=False` past grace).
  - **ArgoCD** `Application` sync-status probe (alternative ingress for the
    same signal that GitOpsDrift watches; the probe is the resource-event
    fast path).
  - **Velero** backup-completion probe (Backup CR phase `Failed` or
    last-successful-backup older than the SLA window).
  - Each probe opt-in via `probes.<name>.enabled` Helm value (default `true`
    when the CRD is present; auto-skip when CRD absent).
  - **Target:** Sprints 2-4. **Effort:** 8 days (2 days per probe).

## Risks & dependencies

  - **R1 — metrics-server availability** (B5): cluster may not have one
    installed. Mitigation: probe gracefully no-ops + emits a single
    `MetricsServerMissing` warning. Don't fail the watcher.
  - **R2 — GCP / Azure auth scaffolding**: IRSA's GCP equivalent is Workload
    Identity Federation; Azure's is AAD Workload Identity. Mitigation: ship
    per-provider `auth.go` adapters mirroring the AWS IRSA pattern; default
    `cloud.gcp.enabled=false` / `cloud.azure.enabled=false`.
  - **R3 — Operator-port shape lock-in**: if Phase 1 ships the CRD
    prematurely, future schema changes become breaking. Mitigation:
    `v1alpha1` API version explicitly; document the no-stability-guarantee
    until `v1beta1`.
  - **R4 — Cloud rate limits**: GCP / Azure API quotas vary by project /
    subscription. Mitigation: per-provider rate budget mirroring the AI
    rate budget (`ai/rate_limit.go`); operator-configurable.
  - **R5 — RBAC ClusterRole bloat**: each drift class + probe needs read
    permissions. Mitigation: keep the additive pattern from v1.7 (each
    analyzer's `enabled=true` extends the Reader ClusterRole via Helm
    template).

## Acceptance criteria

  - **Operator**: `kubectl apply -f ClusterHealthAutopilot` creates a
    watcher Deployment + diagnose CronJob + RBAC matching the spec;
    `kubectl delete` cleans up. Both `Ready` and `WatcherRunning` conditions
    go True.
  - **B4 / B5 / B6**: each ships with synthetic-snapshot tests (matching
    the B1-B3 pattern: 12-15 tests per analyzer); each has
    `analyzers.<name>.enabled` Helm toggle + `CHA_ANALYZER_<NAME>=off`
    env-gate; each is verified on the live cluster within 24h of enable.
  - **M2 K8s probes**: 4 new probes; each has unit tests; each is opt-in
    via `probes.<name>.enabled` Helm value.
  - **GCP / Azure cloud probes**: full parity with AWS — 10 probes per
    provider; same `cloudprobe.Probe` interface; same DriftReport CRD; same
    sinks. Default `cloud.{gcp,azure}.enabled=false`.
  - **CHANGELOG + roadmap**: v1.8.0 entry in OSS CHANGELOG mirroring the
    v1.7.0 PR-level granularity. Website `/roadmap` flipped after release.
